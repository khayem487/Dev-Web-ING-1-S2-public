\section{Introduction}

Ce document presente la realisation du projet de Developpement Web ING1 autour du theme
\textbf{Maison Intelligente}. L'objectif est de livrer une application web complete,
avec architecture full-stack, base de donnees persistante, gestion des roles,
automatisation domotique et module d'administration.

Le projet respecte les attentes du module :
\begin{itemize}
  \item frameworks backend/frontend,
  \item persistance des donnees,
  \item recherche multi-criteres,
  \item separation claire des modules,
  \item demonstration fonctionnelle complete.
\end{itemize}

% ------------------------------------------------------------
\section{Objectifs et perimetre}

\subsection{Objectifs fonctionnels}
\begin{enumerate}
  \item \textbf{Information} : consultation publique des pieces, objets et services.
  \item \textbf{Visualisation} : espace membre (authentification, profil, consultation detaillee).
  \item \textbf{Gestion} : pilotage avance des objets, scenarios, maintenance, statistiques.
  \item \textbf{Administration} : gestion utilisateurs, droits et validation des comptes.
\end{enumerate}

\subsection{Objectifs non-fonctionnels}
\begin{itemize}
  \item interface responsive,
  \item securite d'acces par role et niveau,
  \item traces d'actions,
  \item disponibilite locale et deploiement cloud (Render).
\end{itemize}

\section{Architecture technique}

\subsection{Stack}
\begin{itemize}
  \item \textbf{Backend} : Java 21, Spring Boot 3.3, Spring Data JPA, Validation, Mail SMTP.
  \item \textbf{Frontend} : React 19, Vite.
  \item \textbf{BDD} : H2 (fichier local) et base MySQL prete.
  \item \textbf{Deploiement} : Render (API Docker + Frontend statique).
\end{itemize}

\subsection{Organisation du code}
\begin{itemize}
  \item \texttt{backend/} : modeles, DTO, repositories, services, controllers.
  \item \texttt{frontend/} : pages, composants, etat UI, appels REST.
  \item racine : \texttt{pom.xml}, \texttt{Dockerfile}, \texttt{render.yaml}, \texttt{README.md}.
\end{itemize}

\subsection{Routes REST principales}

  \item \texttt{/api/info/*}

  \item \texttt{/api/visualisation/*}

\section{Securite et controle d'acces}

Le flux de securite applique est le suivant :
\begin{enumerate}
  \item inscription,
  \item verification email par code (envoi SMTP reel),
  \item validation manuelle par administrateur,
  \item connexion autorisee uniquement apres ces etapes.
\end{enumerate}

Un compte non approuve est bloque au login (retour HTTP 403).
Cette approche reduit les inscriptions sauvages et donne un controle operateur
sur les nouveaux comptes.

Les acces aux modules sont verifies cote backend :
\begin{itemize}
  \item module Gestion reserve aux utilisateurs de niveau \texttt{AVANCE},
  \item module Administration reserve aux comptes \texttt{admin=true}.
\end{itemize}

\subsection{Regles d'integrite et de gestion}
\begin{itemize}
  \item Un utilisateur ne peut pas se connecter si \texttt{email\_verifie=false}.
  \item Un utilisateur verifie reste bloque si \texttt{compte\_approuve=false}.
  \item Une action scenario doit viser un objet existant (FK non nulle).
  \item Les suppressions d'objets conservent la tracabilite admin.
  \item Les donnees capteurs sont historisees et jamais ecrasees en masse.
\end{itemize}

\section{Difficultes rencontrees et solutions}

\begin{enumerate}
  \item \textbf{Evolution du schema local H2} :
  gestion des colonnes legacy et compatibilite lors des evolutions modeles.

  \item \textbf{Synchronisation etat front/back} :
  mise a jour locale immediate de l'UI apres reponse API pour eviter la latence percue.

  \item \textbf{Securisation des inscriptions} :
  ajout verification email reelle + validation admin obligatoire avant login.
\end{enumerate}
